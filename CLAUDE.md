
# CLAUDE.md — rdf2tab プロジェクト システムプロンプト

このファイルは、このリポジトリで作業する Claude Code（および将来このプロジェクトを引き継ぐ人間）が **常に読む前提のプロジェクト憲法** である。個別タスクの詳細は `.claude/commands/build-lod-project.md` に書くが、ここに書く原則と決定事項はどのフェーズでも優先される。矛盾があれば、このファイルが勝つ。

## 0. プロジェクトの正体

大学院機械学習科目の最終レポート課題（`final_report-ja.pdf` 参照）として、以下を実装する。

> DBpedia（LOD/RDF）から**任意の RDF スキーマに適用可能な汎用変換フレームワーク**でノード分類用の表形式データセットを構築し、その表形式データに対して **Tabular Foundation Model（TabICLv2 / TabPFNv2）** を適用してノード分類を行い、**LLM API 分類ベースライン**と性能・推論コスト・堅牢性の観点で比較する。RDF→Tabular 変換自体も、複数クラスに適用することで汎用性を実証する。

評価軸（課題 PDF 2ページ目）は次の4つ。**すべての設計判断はこの4軸のどれに効くかを説明できなければならない。**

| 軸 | このプロジェクトでの根拠 |
|---|---|
| 独創性 | LOD/RDF に前例のない（後述の先行研究と明確に差別化された）Tabular FM を適用する。動機・意義を README/報告書で説明する。 |
| 実装の有用性 | 汎用 RDF→Tabular 変換エンジン（設定ファイル駆動、コード変更なしで別クラスに適用可能）を自作実装する。サンプルスクリプトの改変では終わらせない。 |
| 信頼性 | 複数クラス・複数シードでの再現実験、堅牢性評価、統計的な考察に基づいて結論を出す。過大な新規性主張はしない（後述）。 |
| 了解性 | README だけで教員が `pip install` から結果再現まで到達できる。API キーなしでも主要な結果は再現できる（キャッシュ同梱）。 |

## 1. 確定した意思決定ログ（これを覆すときは理由をここに追記すること）

過去の議論・ファクトチェックで確定した事項。実装中に「なぜこうなっているか分からない」となったら、まずここを読む。

1. **対象データセット**: DBpedia のみ（Wikidata は対象外。スキーマの揺れが大きすぎて工数が発散するため）。
2. **対象クラス**: `dbo:Book`（主）と `dbo:Person`（副）の最低2クラス。
   - 理由: 「汎用的な変換ルール」を謳う以上、1クラスだけの実験では「本当に任意のスキーマに使えるのか」を実証できない。ユーザーは当初「DBpedia に限定すると明記すれば1クラスでいいのでは」と提案したが、これはエンドポイントのスコープを狭める話であり、**クラス間のプロパティ集合の異質性という別の問題を解決しない**。`dbo:Book` と `dbo:Film` は属性構造が似すぎて弱い検証にしかならないため、意図的に属性構造が大きく異なる `dbo:Person`（`birthDate`, `nationality`, `occupation` 等、`genre` が存在しない）を選定した。設定ファイルの差し替えだけで動くことが、この変換フレームワークの正しさの証明である。
3. **ラベル**: `dct:subject`（Wikipedia カテゴリ）を上位 `top_k`（既定20、`configs/*.yaml` で調整可）件に制限して使用。`dbo:literaryGenre` のような書籍固有プロパティは `dbo:Person` に存在しないため、クラス非依存で使えるラベルソースとして `dct:subject` を採用した。
   - 上位K件に入らないカテゴリしか持たないインスタンスは既定で **除外**（`unmatched_policy: drop`）。多数派への丸め込み（"other" クラスへの合算）はクラス不均衡を歪めるため既定では行わない。
4. **Tabular Foundation Model**: 当初 Schema-1 (Data Language Model, arXiv:2605.06290, SchemaLabs) を想定していたが、**この論文はエンタープライズ製品の白書であり、公開重みが存在しない**（"Model Runtime" 上で顧客専用チェックポイントを発行する SaaS 型。HuggingFace 等での重み配布を確認できなかった）。**これを主モデルにすると教員側で一切再現できない。** 代わりに以下の**公開重み・pip install 可能**なモデルを採用する。
   - 主: **TabICLv2**（`soda-inria/tabicl`、Apache系ライセンス、in-context learning、公開重み・pip配布あり）
   - 副（クロスチェック用）: **TabPFNv2**（`PriorLabs/TabPFN`、Prior Labs License）
5. **LLM ベースライン**: ユーザーの意向により **ChatGPT API（既定: `gpt-4o-mini`、コスト重視）** を採用する。ただし、これは「教員が追加コストなしで実行できる」という当初目的と真っ向から矛盾する。この矛盾は次の設計で緩和する。**これは交渉の余地がある部分ではなく必須要件**。
   - すべての LLM API 呼び出しレスポンスは `data/processed/llm_cache/` に JSONL でキャッシュし、**Git 管理下に置く**（`.gitignore` の `data/*` 全除外ルールを、このキャッシュ配下だけ明示的に許可するように変更する）。
   - パイプラインは **API キー無し・ネットワーク無しでキャッシュのみを使って全実験を再現できる** ことを必須の受け入れ基準とする（`--offline` フラグ、キー未設定時は自動でこのモードにフォールバックしログで明示）。
   - API キーが必要なのは「新しい LLM 呼び出しを追加で行いたいとき」のみに限定する。
   - 呼び出しごとに prompt/completion トークン数と概算コスト（$）を記録し、評価指標の一部として報告する（「推論コスト」を推論時間・メモリと並ぶ評価軸に追加。これは LLM ベースライン特有の弱点を可視化するので、むしろ分析の見せ場になる）。
6. **古典的 Tabular モデル（XGBoost 等）**: ユーザーの明示判断により**比較対象から除外**し、Tabular FM vs LLM のみで比較する。
   - **これは意図的な弱点として記録する。** GBDT を外すと「Tabular FM が LLM に勝った/負けた」という結果が、「タスクが単に簡単だった」のか「Tabular FM 固有の強みなのか」を切り分ける手段がなくなる。信頼性の評価軸で突かれるリスクを承知の上での判断であり、報告書の「結果と考察」または「限界」節で**この設計上の限界を必ず明記すること**（隠さない・言い訳しない）。
   - ただし、評価ハーネスには**多数派クラス予測ベースライン**（実装コスト数行）を無条件で含める。これは「GBDT を比較対象に含めるか」という議論とは別次元の、指標が意味を持つかどうかの健全性チェックであり、削除しない。
7. **変換処理は CPU のみ**。GPU（ローカル RTX 5070 Ti, 16GB VRAM 想定）は Tabular FM 推論と（ローカル LLM を使う場合の）LLM 推論にのみ使う。
8. **測定する指標**:
   - 推論時間・メモリ（1件推論 / バッチ一括推論の両方、GPU: `torch.cuda.max_memory_allocated`、API: 実測レイテンシ + コスト）
   - 変換時間（SPARQL 抽出時間と RDF→Tabular 変換時間を分離して測定し、サンプル数 N に対するスケーリング曲線を出す。既定 N = 500 / 1000 / 2000 / 5000）
   - 自動化度合いは**定性的に**（どの処理が自動でどれが設定ファイル経由か）記述するのみで、工数を定量化しない。
   - 頑健性 = クラス不均衡耐性。自然な分布と人為的に偏らせた分布の両方で macro-F1 / balanced accuracy を比較する。

## 2. 先行研究との差別化（新規性の主張を守るための必須引用）

「LOD に前例のないモデルを当てはめる」という当初の企画意図は、**無条件には成立しない**。以下の先行研究が存在することを確認済み。**報告書でこれらを引用せずに新規性を主張すると、精査されれば即座に破綻する。**

- **AutoRDF2GML**（ISWC 2024, Färber et al.）— RDF から GNN 用異種グラフを半自動生成する枠組み。本プロジェクトの「対抗馬」的位置づけとして言及する（本プロジェクトは GNN ではなく Tabular FM を使う点で異なる）。
- **RelGNN**（ICML 2025, Stanford SNAP）— リレーショナルDBに対する GNN。本プロジェクトのスコープ外（Relational DL は不採用）だが、関連研究として言及。
- **"Can TabPFN Compete with GNNs for Node Classification via Graph Tabularization?"** および **"Bringing Graphs to the Table: Zero-shot Node Classification via Tabular Foundation Models"**（TabGFM 系）— **Tabular FM をグラフのノード分類に使うこと自体は既に前例がある。** ただし対象は主に構造特徴（隣接情報のテーブル化）中心の一般グラフベンチマーク（引用ネットワーク等）であり、**実世界の異種スキーマ RDF/LOD を、汎用 SPARQL 駆動の変換パイプラインでコンテンツ特徴（リテラル）＋軽量な構造集約特徴に変換した上で Tabular FM に通す、という設定は行っていない**。
  - **新規性の主張はここに絞る**: 「Tabular FM をグラフに使う」のではなく、「**任意の RDF スキーマ・任意の SPARQL エンドポイントに適用可能な汎用変換器を介して、実世界の LOD をノイズを含んだまま Tabular FM に食わせ、LLM API ベースラインと推論コスト・堅牢性の面で比較した最初の実証**」という限定された、しかし具体的で検証可能な新規性を主張する。

## 3. アーキテクチャ原則（非交渉）

1. **設定ファイル駆動（80%自動＋20%設定）**: クラス固有のロジックを Python コードに if/else で埋め込むことを禁止する。`target_class` を変えて `configs/*.yaml` を差し替えるだけで動くこと。これが崩れたら、決定事項 #2 の「汎用性の証明」が嘘になる。
2. **すべての外部呼び出しはキャッシュ必須**: SPARQL エンドポイントへのクエリ、OpenAI API 呼び出しは、すべてディスクキャッシュ（`data/raw/`, `data/processed/llm_cache/`）を経由する。同一クエリを2度打たない。再実行時はデフォルトでキャッシュを使い、`--refresh` 明示時のみ再取得する。
3. **SPARQL は安全側に倒す**:
   - プロパティ URI はホワイトリスト（プレフィックス辞書に定義されたもののみ）で扱う。ユーザー入力を生 SPARQL 文字列に直接埋め込まない。
   - `LIMIT`/`OFFSET` を必ず明示し、無制限 `OPTIONAL` の多重 JOIN を避ける。ページングは 2 段階クエリ（インスタンス取得 → プロパティ取得）に分割する。
   - 公開エンドポイントへの過度な負荷を避けるため、リクエスト間隔・タイムアウト・リトライ（指数バックオフ）を設定可能にする。
4. **メモリディシプリン**: N が大きくなっても全件を一括でメモリに展開しない。ジェネレータ/ストリーミングで行を生成し、Parquet に逐次書き出す。
5. **モジュール境界を守る**: `sparql/`（取得）, `convert/`（変換）, `models/`（推論）, `eval/`（評価）は互いに知りすぎない。`convert` は `models` を import しない。逆も同様。
6. **例外を握りつぶさない**: bare `except:` 禁止。SPARQL タイムアウト、API レート制限、欠損ラベルなどのエッジケースは明示的に型付けした例外/戻り値で扱い、ログに残す。
7. **乱数シード**: Python `random`, `numpy`, `torch` すべてを単一の `set_seed(seed)` で固定する。データ分割・サブサンプリングは常にこの seed を通す。

## 5. 品質基準

- 型ヒント必須（`mypy` 通す前提の書き方。`Any` の濫用禁止）。
- 公開関数・クラスに docstring 必須（何をするかだけでなく、なぜその設計かを一言添える）。
- `print` でのデバッグ禁止、`logging` を使う。
- 各モジュールに対応する `tests/test_*.py` を用意し、**モックした SPARQL レスポンス・モックした API レスポンス**でオフラインテストできること（本物のエンドポイントに毎回叩きに行くテストは CI で禁止）。
- 「テストを後で書く」は認めない。モジュールが動く＝テストがある、が完了の定義。
