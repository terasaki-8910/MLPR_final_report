
# CLAUDE.md — rdf2graph プロジェクト システムプロンプト

このファイルは、このリポジトリで作業する Claude Code（および将来このプロジェクトを引き継ぐ人間）が **常に読む前提のプロジェクト憲法** である。個別タスクの詳細は `.claude/commands/build-lod-project.md` に書くが、ここに書く原則と決定事項はどのフェーズでも優先される。矛盾があれば、このファイルが勝つ。

## 0. プロジェクトの正体

大学院機械学習科目の最終レポート課題として、以下を実装する。

> 図書館・書誌系の実在する SPARQL エンドポイント4件（DBLP, NDL典拠データ, ICCU SBN, Cervantes Virtual）それぞれに対して、**汎用的な RDF→グラフ変換パイプライン**を適用し、**R-GCN によるノード分類**を行う。各エンドポイントは独立したグラフとして扱い（統合しない）、モデルが学習したノード埋め込みに対して**ラベル別平均ベクトル（クラスセントロイド）間のコサイン類似度**を算出することで、スキーマも言語も全く異なる4つの実世界データセット間で「クラスの分離しやすさ」を比較する。

### 0.5 新規性についての正直な自己評価（重要・自己欺瞞禁止）

**R-GCN（Schlichtkrull et al., 2018, "Modeling Relational Data with Graph Convolutional Networks"）は、RDF ナレッジグラフのノード分類に対する最も標準的な手法であり、モデルとしての新規性は一切ない。** 原論文自体が AIFB, MUTAG, BGS, AM という RDF ベンチマークで評価している。これを隠して報告書に「新規性がある」と書くと、GNN や KG を少しでも知っている採点者には即座に見抜かれる。

本プロジェクトが主張してよい新規性は、モデル側ではなく **評価・分析側** に限定する。

- **主張してよいこと**: 「スキーマ・言語・ドメインが大きく異なる実世界の書誌/典拠 LOD 4件（計算機科学文献 DBLP、日本語典拠データ NDL、イタリア書誌 ICCU SBN、スペイン語デジタル図書館 Cervantes Virtual）に対して、コード変更なく設定ファイルの差し替えだけで動く単一の RDF→グラフ変換パイプラインを適用し、同一の R-GCN アーキテクチャでノード分類を行い、学習された埋め込み空間のクラス分離度（クラスセントロイド間コサイン類似度）をエンドポイント横断で比較した」という、地味だが具体的で検証可能な比較分析。
- **主張してはいけないこと**: 「R-GCN を LOD に応用したのは初めて」「新しいモデルを提案した」の類。
- 報告書の「目的」節では、新規性の軸が「比較分析」にあることを明示的に書くこと。

## 1. 確定した意思決定ログ

1. **対象エンドポイント（4件、確定）**: 以下は 16 候補に対して実際に SPARQL ASK クエリを投げて生存確認済み。
   - `https://sparql.dblp.org/sparql` — DBLP（計算機科学文献の書誌データ）
   - `https://id.ndl.go.jp/auth/ndla/sparql` — NDL Web NDL Authorities（日本語の主題・人物典拠データ。**書誌ノードを持たない**ため、分類対象は典拠ノード自体になる）
   - `https://triplestore.iccu.sbn.it/sparql` — ICCU SBN（イタリア国家書誌サービス、実書誌レコード）
   - `https://data.cervantesvirtual.com/sparql` — Cervantes Virtual（スペイン語デジタル図書館、書誌＋伝記データ）
   - 除外理由: GND は NDL と役割（典拠ファイル）が重複するため除外。EU Publications Office は法令オントロジー（CDM）が複雑で時間対効果が悪いため除外。LinkedPapersWithCode は「図書館系」という枠から外れるため除外。
   - **生存未確認の8件（LIBRIS はWAFブロック、SIBiLS はタイムアウト、VU Amsterdam/IdRef/上海図書館/韓国NL/uedayou/Cervantes旧openrdf-sesameエンドポイントは無応答）は採用しない。** これらはこのセッションのサンドボックスからの検証であり、教員の実行環境では挙動が異なる可能性はあるが、確認できないものを前提に設計しない。
2. **グラフ単位**: エンドポイントごとに独立したグラフを構築する。**統合しない。** Ontology alignment・エンティティリンク（owl:sameAs 解決等）は時間的に不可能と判断し、明示的にスコープ外とする。この結果、4エンドポイント間で「同じラベル空間」を共有した比較はできない（決定事項 #3 参照）。
3. **クラス間比較の設計**: ラベル空間を共有しないため、コサイン類似度は **各エンドポイント内で完結**させる。具体的には、学習済み R-GCN のノード埋め込みからラベルごとの平均ベクトル（クラスセントロイド）を計算し、エンドポイント内のクラスペア間コサイン類似度を算出する。これを「そのデータセットにおけるクラスの分離しやすさ」の指標とし、4エンドポイント間で**この指標自体を比較する**（埋め込みベクトルそのものを比較するのではない）。
4. **分類対象ノードとラベル**: エンドポイントごとにスキーマが根本的に異なる（DBLP/ICCU SBN/Cervantes Virtual は書誌データ、NDL は典拠データのみで書誌ノードを持たない）ため、**対象クラス・ラベルプロパティをコード内にハードコードすることを禁止**し、以下の2段階の自動探索で決定する。
   - Phase 2 で「クラスごとのインスタンス数」を集計するスキーマ偵察クエリを実行し、最も代表的なクラスを候補として提示する。
   - そのクラスに対して述語プロファイラを実行し、被覆率の高いプロパティからラベル候補・特徴量候補を提示する。
   - **これらの述語URIを事前に決め打ちしない。** DBLP・ICCU SBN・Cervantes Virtual の正確なオントロジーURIは未調査であり、実装フェーズでの実地確認が必須。
5. **ノード特徴量**: 構造特徴（次数、リレーションタイプ別隣接数）に加えて、タイトル・ラベル文字列に対する**浅いテキスト特徴（TF-IDF または hashing vectorizer、低次元）**を使う。BERT等の深い埋め込みは使わない（LLM+GNN融合領域には踏み込まない）。TF-IDF語彙はエンドポイントごとに独立に学習する（言語が英語・日本語・イタリア語・スペイン語とバラバラなため、共有語彙は意味を持たない）。
6. **規模上限**: エンドポイントあたり **ノード数 10,000〜20,000、エッジ数 10万程度**を上限とする。SPARQL の `LIMIT`/`OFFSET` で明示的に制御する。
7. **モデル**: R-GCN（2〜3層、`torch_geometric.nn.RGCNConv` を使用）。学習・推論は GPU（RTX 5070 Ti）。GPU が無い環境では CPU フォールバックすること（教員の環境を考慮）。SPARQL 抽出・グラフ構築は CPU のみ。
8. **ベースライン（2本、無条件で実装・議論の対象にしない）**:
   - 多数派クラス予測ベースライン（数行）。
   - 特徴量のみ MLP/ロジスティック回帰ベースライン（グラフ構造を一切使わず、ノード自身の特徴量だけで分類）。**これが無いと「R-GCNの性能向上がグラフ構造由来なのか、単にノード特徴が強いだけなのか」を切り分けられない。** ここは削らない。
9. **評価軸**:
   - 精度: macro-F1, balanced accuracy。
   - コスト: 学習時間・推論時間・GPU/CPUメモリ（1エンドポイントあたり）。
   - 変換時間: SPARQL抽出時間とグラフ構築時間を分離して測定し、ノード数に対するスケーリング傾向を記録する（CPU条件を固定）。
   - **クラス分離度**: ラベル別クラスセントロイド間コサイン類似度（決定事項 #3）。
   - **意図的に含めないもの（時間制約による）**: グラフ構造ロバスト性（エッジ削除・ノイズ耐性）、スケールアウト性、クラス不均衡を人為的に操作した頑健性評価。**これは意図的な弱点であり、報告書の限界節で明記すること。** 隠さない。
10. **外部依存を持たない**: LLM APIや商用Tabular Foundation Modelのような、課金・APIキーを要する外部サービスには一切依存しない。公開SPARQLエンドポイントとローカルGPUのみで完結する。教員が追加コストなしで再現できるという点を報告書に書いてよい。
11. **仮想環境（`ml_env/`）は完全独立とし、`--system-site-packages`を使わない**: 一度は「開発機に既に動作確認済みのtorch+CUDAがあるから流用する」設計で`--system-site-packages`付きvenvを提案したが、これは決定事項#10（教員環境での再現性）と直接矛盾すると指摘され撤回した。教員のマシンでは`--system-site-packages`が既存のグローバル環境（バージョン違い・CPU版・あるいは何も無い状態）をそのまま素通しし、requirements.txtの`torch`が無バージョン指定である以上pipがインストールをスキップしうるため、「動くかどうかがそのマシンの既存グローバル状態に依存する」という最も避けたい状態に逆戻りする。開発機ではCUDA版torchの再ダウンロードコスト（数GB）を許容し、`python -m venv ml_env`（フラグなし）→ `pip install torch==2.11.0 --index-url https://download.pytorch.org/whl/cu128` → `pip install -e .` の順で構築する。requirements.txt自体は`torch`のビルド指定をしないままにし、教員のGPU無し環境ではPyPI既定のCPUビルドがフォールバックとして入るようにする。
12. **torch-scatter / torch-sparse / pyg-lib は意図的に含めない**: これらはtorch/CUDAの組み合わせごとのビルド済みwheel配布に依存し、新しいtorchリリース（本プロジェクトでは2.11.0）に対応するwheelが無い場合、Windows上でのソースビルドが不安定になりやすい。`torch_geometric>=2.5`の`RGCNConv`は純PyTorchフォールバックで動作する。この判断は「importが通ること」だけでなく、`tests/test_rgcn_smoke.py`で本プロジェクトの想定規模（ノード15,000/エッジ100,000/リレーション15種類相当）のforward/backwardを実際に通し、CPU・CUDA（RTX 5070 Ti）の両方で数秒以内に成功することを実地検証済み（2026-07-06）。このテストが将来失敗する、または実用上遅すぎると判明した場合はtorch-scatter/torch-sparseの追加導入を再検討する。
13. **SPARQLClientの例外階層とパース経路の統一**: `SPARQLQueryError`（4xx等、リトライしない）/`SPARQLTimeoutError`（リトライ上限超過）/`SPARQLResponseFormatError`（Content-Type不一致・JSONデコード失敗）を型として区別する。`query()`はSELECT系ならBindingの行リスト、ASK系ならboolを返し、`paginate()`は`query()`を介して1ページずつ取得するだけで独自のパース処理を持たない（`_parse_bindings`に一本化）。キャッシュファイルには元のクエリ文字列を保存し、読み込み時に不一致を検知したらWARNINGログを出してミス扱いにする（ハッシュ衝突・書き込み破損の両方に対する保険）。
14. **ページング終了条件とサーバー側result capの区別は未解決（緩和策のみ）**: `paginate()`は「ページが`page_size`未満なら終了」で判定するが、これは真の終端とサーバー側の暗黙のresult capを区別できず、WARNINGログによる可視化のみの緩和策である。**これは仮説ではなく実証済みの制約**: `scripts/check_endpoints.py`による実地検証で、NDL Web NDL Authoritiesが`LIMIT 5000`要求に対し`1000`行しか返さないことを確認した（`docs/endpoint_status.md`参照、2026-07-06実施）。Phase 3で`configs/ndl_authorities.yaml`の`sampling.page_size`を確定する際、1000を超える値は無言で切り詰められる前提で設計すること。
15. **Phase 4のグラフ表現は`HeteroData`ではなく`edge_type`付き単一`Data`を採用**: Phase 5で使用する`RGCNConv`は`edge_type`テンソル付きの単一`Data`オブジェクトを直接消費する設計であり、これはPyGのAIFB/MUTAG等RDFノード分類例と同じパターンである。HeteroDataを使うとRGCNConvとの間に追加の変換層が必要になり複雑化するだけでメリットがない。これはPhase 2の発見結果に依存しないアーキテクチャ選択であり、決定事項#4（URI決め打ち禁止）の対象外として先行確定した。
16. **Phase 2 スキーマ偵察の結果、「インスタンス数上位=分類対象として適切」ではないことが実証された（決定事項#4の裏付け）**: `rdf2graph/sparql/profiler.py`の`discover_classes`を4エンドポイントに対して実行した結果（`docs/profile_*.md`）、DBLPとICCU SBNでは**インスタンス数最上位のクラスが、書誌レコードそのものではなく支援的・管理的なクラスだった**。
    - DBLP: 1位は`cito:Citation`（1.59億件、引用グラフの各引用エッジをノード化したもの）。書誌レコードに相当する`dblp:Publication`は8,625,948件で6位、`dblp:Inproceedings`/`dblp:Article`はさらに下位。
    - ICCU SBN: 1位は`bibframe:Item`（2,996万件、個々の所蔵館ごとの実物資料）、2位は`bibframe:Title`（付随エンティティ）。書誌レコードに相当し得る`bibframe:Work`（879万件・13位）や`bibframe:Instance`（1,121万件・7位）はさらに下位。
    - また ICCU SBN では `bibframe:Local`/`bibframe:GenerationProcess`/`bibframe:AdminMetadata` の3クラスが偶然ではなく**完全に同一のインスタンス数（9,420,045件）**を示した。丸い数字ではないため`discover_classes`の疑わしさ判定（決定事項#14）には引っかからないが、これは`bibframe:Instance`1件につき機械的に1つずつ生成される管理用メタデータノード群である可能性が高く、サーバー側result capの兆候ではないと判断した（Phase 3で`target_class`候補から外すため実害はない）。
    - NDLとCervantes Virtualでは、インスタンス数上位クラスがそのまま自然な分類対象候補（NDL: `skos:Concept`/`foaf:Person`/`foaf:Organization`＝典拠ノード自体、決定事項#1参照。Cervantes Virtual: `rdaregistry:Work`/`Expression`/`Manifestation`＝FRBR的書誌モデル）と一致した。
    - この非対称性（4エンドポイント中2件で「1位を鵜呑みにすると支援クラスを掴む」）は、決定事項#4が「クラス・ラベルプロパティの決め打ち禁止」を定めた根拠を実地で裏付けるものであり、Phase 3の`target_class`確定は`docs/profile_*.md`の上位数件を人間が精査した上で行う（機械的な1位選択を採用しない）。
17. **`suggest_label_candidates`の初版スコアリングは欠陥があり、レビューで指摘されて再設計した**: 初版は「被覆率が高くliteralが多い」ことだけを基準にしており、(a) `skos:inScheme`のようなURI型の閉じた集合述語を型がuriというだけで無条件除外し、(b) `rdfs:label`/`dc:title`/`dc:created`のような自由テキスト・タイムスタンプ（1インスタンス1値でほぼ一意）を被覆率が高いという理由だけで最上位に誤って提案する、という2つの欠陥があった。再設計後は「値の異なり数(カーディナリティ)が値を持つ主語数に対してどれだけ小さいか」を主基準にし、`literal`/`uri`混在述語（例: `dc:subject`がLCSH文字列と人物URIを同居させるケース）は`literal`側のみを判定対象にする。また、異なり値数が1（＝実質定数値、例: Cervantes Virtualの`dc:terms:created`が全インスタンスで収穫日時スタンプ"02/02/2026"固定）の述語はカーディナリティ比が最小になり素朴なソートでは最上位に来てしまうため、`min_distinct_values=2`で明示的に除外する。`suggest_feature_candidates`（特徴量候補）は今回の対象外（自由テキストでもハッシュ化/TF-IDF化すれば特徴量として有効なため、カーディナリティで絞り込まない）。
18. **Phase 2 述語プロファイラは、上位N件の自動選定に加えて「本命候補」を人間が個別指定して深掘りできるようにした**: 決定事項#16の教訓（インスタンス数1位が支援クラスであるケースがある）を活かすには、上位ランクだけでなく人間が疑わしいと判断したクラスも深掘りする必要がある。`scripts/run_profiler.py`の`EXTRA_CLASSES_OF_INTEREST`にエンドポイントキーごとの追加クラスURIを列挙し、`discover_classes`の上位20件から実インスタンス数を引いた上で`profile_predicates`に渡す（`profile_predicates`に`total_instance_count`引数を追加し、サンプル件数とは別に真の総数をMarkdownに表示できるようにした）。DBLPの`dblp:Publication`、ICCU SBNの`bibframe:Work`/`bibframe:Instance`をこの機構で追加プロファイルした。
19. **重要な未解決の懸念（隠さず明記）: `profile_predicates`のサンプリングがランダムでない可能性が実地で示唆された**: DBLPの`dblp:Publication`（全体8,625,948件）を200件サンプリングした結果、`rdf:type`の内訳が`Incollection`188件（94%）、`Editorship`9件、`Book`3件のみで、`Inproceedings`(全体3,910,938件=約45%相当)や`Article`(3,381,776件=約39%相当)が1件も現れなかった。`discover_classes`で得た各サブタイプの全体件数から期待される比率（Inproceedings約45%・Article約39%・Informal約11%・Incollection約0.5%）とはかけ離れており、`profile_predicates`が使う「サブクエリ+LIMIT」方式のサンプリングが、エンドポイント側の物理的な格納順・挿入順に強く相関した非ランダムな結果を返している疑いが強い。これは`discover_classes`のGROUP BY集計とは別種の問題（result capではなくサンプリングバイアス）であり、Phase 2で提案する「ラベル候補」「特徴量候補」のカーディナリティ推定値そのものの信頼性に影響しうる（本来は多様な値を取る述語が、非ランダムサンプルでは見かけ上「閉じた集合」に見えてしまう可能性がある）。Phase 3で`configs/*.yaml`を確定する際、または将来Phase 4のノード抽出時にランダムサンプリング（例: ハッシュベースの疑似ランダムOFFSETや`ORDER BY RAND()`、対応エンドポイントでの`SAMPLE`系機能の利用）を導入するかどうかは、ユーザーに判断を仰ぐ。現時点のPhase 2出力はこの限界を踏まえ「傾向を掴むための参考情報」として扱い、確定判断の唯一の根拠にしないこと。
20. **ラベルプロパティの値分布は行単位サンプリングを経由せず、GROUP BY全数集計で直接検証する（決定事項#19の懸念に対する採用解）**: ランダムサンプリング機構の新規実装（`ORDER BY RAND()`は大規模クラスでサーバー側フルソートが必要でタイムアウトの危険）は採らず、`discover_classes`と同じGROUP BY集計パターン（`rdf2graph/sparql/profiler.py`の`count_label_values`、`scripts/run_label_census.py`）で対象クラス全数に対するラベル候補値の真の分布を取得した（2026-07-06実施、各`docs/profile_*.md`の「全数集計によるラベル候補検証」節）。この結果:
    - **決定事項#19のサンプリングバイアスは疑いから実証に変わった**。DBLPの`dblp:bibtexType`の真の分布は`Article`50.7%・`Inproceedings`45.4%（2値で96%）だが、200件サンプルでは`Incollection`94%で`Article`/`Inproceedings`が1件も現れなかった。200件サンプルの値分布は全数分布と全く別物であり、`profile_predicates`のサンプル統計を分布推定に使ってはならない（傾向把握・述語の存在確認のみに使う）。
    - **DBLP・NDLはラベルとして健全**: DBLPの`bibtexType`は8値の閉じた集合（被覆100%）。NDLの`skos:inScheme`は10値の閉じた集合（被覆100%、最頻値`personalNames`70.3%）。この2件は`configs/dblp.yaml`・`configs/ndl_authorities.yaml`を確定した。
    - **ICCU SBNの`bibframe:subject`にはエイリアスTopic URI問題がある（実証済み）**: 全数集計で件数が完全一致するTopicペア（例: `convegnicongressi-`と`congressi-e-convegnicongressi-`が共に20,539件）が多数あり、共起カウントで「同じWork集合に両方付与される異表記エイリアス」であることを確認した（共起数=各件数が完全一致）。また上位50値の合計でもWork全体の5.1%しか被覆しない（サンプルで36%に見えた被覆率は「何らかのsubjectを持つ率」であり、上位K値への被覆率はさらに桁違いに低い）。ユーザーが想定した「drop方式で抽出母数を2.5〜3倍」は成立しない（実際は〜50倍必要）。
    - **Cervantes Virtualの`dc:subject`(literal)は最頻値が空文字列**（3,816件）で、`;`連結の複合主題文字列も多く、上位50値の合計被覆はWork全体の15%。同じくdrop方式の母数想定が成立しない（〜16倍必要）。
    - この2件（ICCU SBN・Cervantes Virtual）は「configを確定する前に再度報告しろ」というユーザーの指示（分布が想定より極端に歪んだ場合のゲート）に該当するため、config確定を保留してユーザーの判断を仰いだ。

## 2. 先行研究との関係（正直な記述必須）

- **R-GCN 原論文**（Schlichtkrull et al., 2018）: 本プロジェクトが採用するモデルそのもの。RDFナレッジグラフのノード分類に対する標準手法であることを明記し、モデル新規性を主張しない根拠として引用する。
- 新規性の主張は「0.5」節の通り、評価・分析側（複数の異種スキーマ実世界LODへの同一パイプライン適用＋クラス分離度のエンドポイント横断比較）に限定する。
- **AutoRDF2GML**（ISWC 2024, Färber et al.）は関連研究として言及する。RDF→異種グラフ変換という点で最も近い先行研究であり、比較対象としてではなく「本プロジェクトが簡易版として何を諦めたか」を説明する文脈で使う。

## 3. アーキテクチャ原則（非交渉）

1. **設定ファイル駆動（80%自動＋20%設定）**: エンドポイント固有のロジックを Python コードに if/else で埋め込むことを禁止する。`configs/*.yaml` を差し替えるだけで新しいエンドポイントに対応できること。
2. **すべての外部呼び出しはキャッシュ必須**: SPARQL クエリ結果は `data/raw/` にキャッシュする。同一クエリを2度打たない。
3. **SPARQL は安全側に倒す**: プロパティURIのホワイトリスト管理、`LIMIT`/`OFFSET`の明示、2段階クエリ（クラス発見→インスタンス取得→プロパティ取得）、リトライ・バックオフ・タイムアウト設定。
4. **メモリディシプリン**: ノード数上限は決定事項#6で明示的に制御されているため厳密なストリーミングは必須ではないが、エッジ収集はジェネレータベースで行い、不要な中間コピーを作らない。
5. **モジュール境界を守る**: `sparql/`（取得）, `convert/`（グラフ構築）, `models/`（R-GCN・ベースライン）, `eval/`（評価）は互いに知りすぎない。
6. **例外を握りつぶさない**: bare `except:` 禁止。SPARQLタイムアウト、スキーマ発見失敗、ラベル欠損などは型付けした例外/戻り値で扱う。
7. **乱数シード**: Python `random`, `numpy`, `torch` すべてを単一の `set_seed(seed)` で固定する。

## 7. デバッグ・失敗時の運用ルール

- **失敗時の挙動**: `tests/` が失敗した場合、(1) 失敗ログを要約し原因候補を列挙する、(2) 1つの修正案をパッチとして提示する、(3) ユーザーの承認を待たずに再テストまで実行してよい（ローカル完結で副作用がないため）。**同一フェーズで3回以上連続して修正が失敗したら、そこで手を止めてユーザーに報告する。** 無限に自己修正ループを回さない。
- **人間の承認が必須な操作**: 公開SPARQLエンドポイントに対する大規模クエリ（`configs/*.yaml` の `max_nodes`/`max_edges` 上限を超える抽出）、GPU学習ジョブの開始（Phase 5以降）は、事前に実行計画を提示してからユーザーの承認を待つ。実行コスト・時間が大きく、後戻りしにくいため。
- **ログ**: 各フェーズの作業は Claude Code の `--debug-file` オプションで保存することを推奨する（触ったファイル・実行したコマンド・テスト結果を後から追跡できる）。
- **このプロジェクト専用の MCP サーバーを自作しない**: フェーズ別テスト実行・プロファイル要約・グラフ統計取得などをMCPサーバー化することを禁止する。これらは `scripts/` 配下の素の Python スクリプトとして実装すれば、Claude Code が直接 Bash で叩けば済む。MCPサーバーはプロトコル層・プロセス管理・自身のバグという新しい失敗表面を追加するだけで、単一開発者・単一マシンのこのプロジェクトでは得られる価値がコストに見合わない。同じ理由で、Sentry等の外部監視サービスやプロンプトテンプレート専用MCPサーバーの導入も却下する（決定事項#10「外部依存を持たない」に反する）。
