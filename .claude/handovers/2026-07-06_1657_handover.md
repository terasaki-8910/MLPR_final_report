## 決定事項・方針

（詳細は `CLAUDE.md` 「1. 確定した意思決定ログ」#11〜#15 と「7. デバッグ・失敗時の運用ルール」を参照。要点のみ）

- **`ml_env/` は完全独立venv。`--system-site-packages` は使わない。** 教員環境での再現性（決定事項#10）を優先し、開発機での再ダウンロードコストを許容する（決定事項#11）。当初 `--system-site-packages` で提案し却下された経緯あり。
- venv構築順序: `python -m venv ml_env` → `pip install torch==2.11.0 --index-url https://download.pytorch.org/whl/cu128`（**バージョンを明示的に固定**。無指定だと「動作確認済みの組み合わせを再現する」という意図と矛盾すると指摘された）→ `pip install -e .`。
- **torch-scatter / torch-sparse / pyg-lib は導入しない。** `torch_geometric>=2.5` の純PyTorchフォールバックで代替。「importが通ること」だけでなく `tests/test_rgcn_smoke.py` で本番規模（ノード15,000/エッジ100,000/リレーション15種）のforward/backwardを実地検証済み（決定事項#12）。
- **SPARQLClient設計**（決定事項#13）:
  - `query()` はSELECT系ならBindingの行リスト、ASK系ならboolを返す。
  - `paginate()` は `query()` を介するだけで独自パース処理を持たない（`_parse_bindings` に一本化）。
  - 例外階層: `SPARQLQueryError`（4xx、リトライしない）/ `SPARQLTimeoutError`（リトライ上限超過）/ `SPARQLResponseFormatError`（Content-Type不一致・JSONデコード失敗）。
  - キャッシュファイルに元クエリ文字列を保存し読み込み時に照合（ハッシュ衝突・破損対策）。
- **ページング終了条件（`page_size`未満で終端とみなす）はサーバー側result capと区別できず、WARNINGログのみの緩和策。根本解決ではないことを承認済み**（決定事項#14）。
- **Phase 4は`HeteroData`ではなく`edge_type`付き単一`Data`を採用**（RGCNConvが直接消費できる設計、決定事項#15）。Phase 2の発見結果に依存しないため決め打ち禁止（決定事項#4）の対象外として先行確定した。
- **CLAUDE.md「7. デバッグ・失敗時の運用ルール」を新設**（別セッションでの決定を一字一句そのまま追記、要約・言い換え禁止という指示だった）:
  - テスト失敗時は要約→修正案提示→ユーザー承認を待たず再テストまで可。ただし同一フェーズで3回連続失敗したら手を止めて報告。
  - 大規模SPARQLクエリ・GPU学習ジョブ開始は事前承認必須。
  - プロジェクト専用MCPサーバーは自作禁止（`scripts/`配下の素のPythonスクリプトで十分。決定事項#10「外部依存を持たない」と矛盾するため）。

## 注意点・ハマりどころ・前提条件

- **このセッションは教員環境ではなくユーザー本人の実機（Windows, RTX 5070 Ti）で動作している。** サンドボックス越しの確認ではないため疎通確認・GPU検証の結果はそのまま信頼してよいが、教員のマシンでの再現性は別途保証が必要（だからこそvenvを完全独立にした）。
- **`SPARQLWrapper` の `QueryResult.info()` が返す `KeyCaseInsensitiveDict` に罠がある**: `__getitem__`（ブラケットアクセス）はキーを小文字化して比較するが、`dict.get()` はオーバーライドされておらず大文字小文字を区別する。`.info().get("Content-Type", ...)` という書き方は実機4エンドポイント全てで空文字を返し、Content-Type検証が常に失敗する不具合を実際に踏んだ。修正済み（`_content_type_of` はブラケットアクセスのみ使用）だが、SPARQLWrapper周りで同種のコードを書く際は要注意。
- **NDL Web NDL Authoritiesは`LIMIT 5000`を要求しても`1000`行しか返さない**（実証済み、`docs/endpoint_status.md`参照）。サーバー側result capの実例。
- **ユニットテストのモックは実際の依存ライブラリの挙動を単純化しすぎる危険がある** — 上記のKeyCaseInsensitiveDictの不具合は、モックが素朴な`dict`を使っていたため検出できず、実エンドポイントへの疎通確認（`scripts/check_endpoints.py`の実行）で初めて発覚した。Phase 2以降もモックテストの成功を過信せず、可能な範囲で実データ確認を省略しないこと。
- **venv構築の順序が重要**: `requirements.txt`では`torch`を無バージョン・無ビルド指定のままにし、開発機では先にCUDA版torchを明示バージョン指定でインストールしてから`pip install -e .`を実行する。逆順や`--system-site-packages`は教員環境での再現性を壊す。
- `.gitignore`に`ml_env/`と`*.egg-info/`を追加済み。`data/*`は既存設定で除外済み。
- `pytest.ini`で`slow`マーカーを登録済み。デフォルトの`pytest`実行では`slow`マーカー付きテストも実行される（除外していない）ため、Phase 0完了定義（`pip install -e . && pytest`が通ること）は維持されている。
- **Windows + git-bash環境**: venv有効化は `source ml_env/Scripts/activate`（PowerShellなら`ml_env\Scripts\Activate.ps1`）。
- 本セッションのPhase 0-1プランは却下と再提出を3往復した。最終承認版の全文は `C:\Users\e8910\.claude\plans\parsed-petting-curry.md` に残っているが、**セッション固有の一時ファイルであり将来消える可能性がある**。内容は`CLAUDE.md`と`docs/progress.md`に反映済みなので、参照する必要が生じてもそちらを優先すること。
